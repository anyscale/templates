Dynamic tokenization stacking directly targets the main weakness of the NVIDIA Build Your Own Transaction Model Blueprint: sequence bloat and loss of intra-transaction alignment. [1, 2] 
By treating each attribute (Amount, Merchant ID, Time Delta, Location) as its own independent token, NVIDIA's standard recipe converts a history of 100 transactions into a sequence of 600+ tokens. This heavily degrades self-attention efficiency, destroys structural alignment, and artificially expands context windows. [1, 3, 4] 
Moving to a stacked architecture where you collapse tabular attributes into unified, dynamic transaction-level vectors offers several distinct design advantages.
1. The Tokenization Stacking Architecture
Instead of flattening a single multi-attribute transaction into an elongated chronological line of text-like tokens, stack them vertically first, then pass them linearly through time.
Attribute Layer Encoders: Run continuous variables (like amount) through a learned projection layer (MLP or Periodic Activation functions). Run categorical variables (like merchant_id or mcc) through separate categorical embedding tables. [5] 
The "Stacking" Aggregator: Combine these aligned attribute embeddings for a single transaction. Instead of simple concatenation (which explodes dimensionality), use a multi-head Intra-Transaction Attention layer or an attention-pooled fusion network. This fuses N attribute vectors into a single, comprehensive "Transaction Token."
The Temporal Backbone: Feed this single sequence of transaction-level tokens into your main causal or masked transformer backbone (e.g., Llama or Mistral architecture). [6, 7] 
2. How to Make It "Dynamic" to Beat NVIDIA
To achieve a competitive edge, the stacking cannot be static. Real transaction logs are inherently heterogeneous: a point-of-sale card transaction has different fields than an ACH wire transfer or a peer-to-peer digital wallet payment. A fixed stacking layer breaks when fields are null or missing.
To outperform the native blueprint, implement these dynamic techniques:
Dynamic Masked Tabular Pooling: Train a lightweight hypernetwork or attention gating layer that dynamically weights fields depending on what is populated. If a transaction lacks a geolocation token but has an IP address token, the stacking aggregator re-weights its cross-attribute pooling matrices on the fly. [4, 7] 
Learnable BPE-Style Subword Merging for Tables: Instead of mapping fields 1:1, look at Dynamic Tokenization Strategies that apply on-the-fly byte-pair encoding concepts to tabular sequences. High-frequency, highly repetitive attribute combinations (e.g., a specific low amount + specific recurring online subscription merchant ID) can be dynamically compressed into a single dense token, while rare anomalies are broken out into high-granularity attribute stacks. [4, 7] 
3. Key Optimization Metrics
Metric / Dimension	NVIDIA "Token per Attribute"	Your "Dynamic Tokenization Stacking"
Sequence Length	L × M tokens (Transactions × Attributes)	Exactly L tokens (1 token per transaction)
Attention Overheads	Quadratic complexity grows $O((L \cdot M)^2)$	Quadratic complexity stays low at O(L²)
Intra-Tx Structural Context	Model must learn via causal self-attention that Attribute A and Attribute B belong to the same point in time.	Structurally guaranteed. The stacking network enforces intra-transaction relationships before temporal processing.
Inference Latency	High memory bound during KV-cache generation.	Fast. Maximizes compute-bound prefill phase and minimizes decode steps.
4. Code Implementation Concept
This PyTorch workflow outlines how to stack features into unified tokens before passing them to the main sequence transformer, bypassing the need to flatten attributes linearly.
import torch
import torch.nn as nn

class DynamicStackingTokenizer(nn.Module):
    def __init__(self, embed_dim, num_categories, num_numerical):
        super().__init__()
        # 1. Independent attribute embedders
        self.cat_embedder = nn.Embedding(num_categories, embed_dim)
        self.num_projector = nn.Linear(num_numerical, embed_dim)
        
        # 2. Intra-transaction attention (Fuses the stacked attributes)
        self.intra_tx_attention = nn.MultiheadAttention(embed_dim, num_heads=4, batch_first=True)
        self.layer_norm = nn.LayerNorm(embed_dim)

    def forward(self, cat_features, num_features):
        # cat_features: [batch, seq_len, num_cats]
        # num_features: [batch, seq_len, num_nums]
        B, L, _ = cat_features.shape
        
        # Embed attributes
        c_emb = self.cat_embedder(cat_features) # [B, L, num_cats, d]
        n_emb = self.num_projector(num_features).unsqueeze(2) # [B, L, 1, d]
        
        # Combine into an attribute stack per transaction
        tx_stack = torch.cat([c_emb, n_emb], dim=2) # [B, L, Total_Attributes, d]
        
        # Collapse dimensions to perform intra-transaction pooling
        # Flatten batch and sequence to handle intra-attention independently per transaction
        tx_stack_flat = tx_stack.view(B * L, -1, tx_stack.size(-1)) # [B*L, Total_Attributes, d]
        
        # Self-attention over attributes belonging to the same transaction
        attn_out, _ = self.intra_tx_attention(tx_stack_flat, tx_stack_flat, tx_stack_flat)
        
        # Pool across attributes (e.g., mean pool) to get 1 Token per Transaction
        tx_tokens_flat = attn_out.mean(dim=1) # [B*L, d]
        tx_tokens = tx_tokens_flat.view(B, L, -1) # [B, L, d]
        
        return self.layer_norm(tx_tokens) # Forward this straight into your Llama/Mistral backbone
Challenges to Anticipate
Vanishing Gradient Bottleneck: Because gradients have to backpropagate through the temporal backbone and then down through the intra-transaction stacking attention layer, you risk bottlenecking your token embeddings. Use strict layer normalization and residual connections directly around the pooling/stacking mechanism.
Pre-Training Complexity: The standard NVIDIA approach allows for direct token mask-filling (Predict the missing token string). With stacked tokens, your pre-training task changes. You either need to use a Contrastive Learning objective (maximizing mutual information between true history and future transaction embeddings) or a Multi-Task Regression/Classification Head that unpacks the output token back into its original component attributes to compute a loss. [8, 9]

To beat the flattened attribute sequence method, your architecture needs to handle three distinct dimensions of data simultaneously: within the transaction (intra-transaction structural context), across time (inter-transaction historical sequence), and variable feature schema (handling missing or transaction-specific fields).
Here is a comprehensive breakdown of the architecture, the optimal feature types, and a specialized positional encoding scheme tailored for dynamic tokenization stacking.
1. The Multi-Tiered Architecture Blueprint
To keep the temporal transformer backbone computationally light, offload feature processing to a localized, isolated sub-network. This isolates the heavy quadratic calculations (O(L²)) strictly to the transaction sequence length (L), while keeping attribute fusion localized to a small, fixed matrix.
[ Raw Tabular Schema ] -> (Categorical, Continuous, Graph, Text, Time)
                                    │
                                    ▼
[ Alignment Layer ]     -> (Linear projections & Embedding Lookups to 'd' dim)
                                    │
                                    ▼
                       ┌─────────────────────────┐
                       │  Dynamic Attribute      │ <-- Masks out null/unpopulated 
                       │  Stacking Block         │     fields dynamically per tx.
                       └─────────────────────────┘
                                    │  (Attention Pooling or Gated MLP Fusion)
                                    ▼
[ Tokenized Sequence ] -> [ Tx_Token 1, Tx_Token 2, ..., Tx_Token L ] (1 token per tx)
                                    │
                                    ┼ <-- Add Complex Dual-Scale Positional Encoding
                                    ▼
[ Temporal Backbone ]  -> [ CaLM / Decoder-Only Transformer (e.g., Llama-style) ]
2. The Features: What Goes in the Stack?
In fraud detection, feeding raw data into a model yields poor results; you must capture multiple layers of context. A dynamic stacking model should group your feature stack into five core categories per transaction:
Core Event Data: Card presence indicator, transaction amount (log-scaled), entry mode (chip, contactless, online), and fallback flags.
Entity & Behavioral ID Embeddings: Cardholder ID, Merchant ID, Merchant Category Code (MCC), IP Address, Device ID, and Zip/Postal Code.
Graph/Topology Structural Context: Instead of raw IDs, run a real-time Graph Neural Network (GNN) over your entity-link graph. Extract node embeddings for the user and merchant nodes and pass these dense vectors directly into the transaction stack. This injects local ring-fraud topology into the sequence.
The Temporal Deltas: Time elapsed since the last transaction on this account, time elapsed since the last transaction on this specific device, and time elapsed since the last transaction at this specific merchant.
Enriched Payload Embeddings: If dealing with ISO 20022 wire text messages or raw device fingerprint user-agents, pass them through a lightweight frozen encoder (like a small BERT or CLIP-text equivalent) to generate a dense sub-vector to include in the stack.
3. Dynamic Masking for Variable Schemas
Real transaction streams have missing fields (e.g., an in-store chip purchase has no IP address, an online purchase has no terminal ID).
Instead of padding missing attributes with zero vectors—which dilutes the attention pooling layer—use Gated Attention Stacking. Create a binary mask matrix $M \in \{0, 1\}^{B \times L \times A}$ representing whether feature A is present. Multiply the attention weights inside your intra-transaction attention block by this mask, forcing the stacking layer to dynamically redistribute its pooling weights over only the populated attributes.
4. The Positional Encoding Strategy (The Secret Sauce)
Standard transformers use RoPE (Rotary Position Embeddings) or learned 1D encodings that assume uniform spacing (Step 1, Step 2, Step 3). This fails in fraud detection. [1, 2, 3] 
A cardholder might make 3 transactions in 2 minutes, then wait 40 days before making a 4th transaction. A standard 1D positional index treats the step from 3 to 4 the same as the steps from 1 to 2, wiping out critical velocity signals.
To fix this, implement a Dual-Scale Positional Encoding:
Scale A: Intra-Transaction Attribute Bias (Learned)
Within the stacking layer, the model needs to know what each vector represents. Assign a learned, fixed positional embedding to each attribute slot (e.g., Position 0 = Amount, Position 1 = Merchant ID, Position 2 = GNN Vector). Add this before running intra-transaction pooling.
Scale B: Inter-Transaction Continuous Time Encodings (Temporal)
For the main temporal transformer backbone, replace standard tokens with Continuous Time-Delta Embeddings using Bochner's Theorem (sine/cosine projections of time differences).
Instead of mapping a transaction to an index t, map the absolute timestamp difference $\Delta \tau = \tau_i - \tau_{i-1}$using a bank of learned frequencies:
$$\Phi(\Delta \tau) = \left[ \sin(\omega_1 \Delta \tau), \cos(\omega_1 \Delta \tau), \dots, \sin(\omega_k \Delta \tau), \cos(\omega_k \Delta \tau) \right]$$ 
To implement this effectively alongside modern architectures, inject this temporal signal into a Time-Aware Rotary Position Embedding (T-RoPE). Instead of advancing the rotation angle uniformly by +1 for each token, advance the rotation matrix angle as a function of the actual wall-clock time passed since the first transaction in the user’s history window. This approach allows the self-attention mechanism to naturally decay its dependency on old transactions based on actual time elapsed, rather than token count. [4]

This framework integrates dynamic categorical/numerical feature separation, gated masking for missing fields, intra-transaction attention pooling, and a downstream temporal sequence processor.
import torch
import torch.nn as nn
import math

class TimeAwareRotaryEmbedding(nn.Module):
    def __init__(self, dim, max_wavelength=10000):
        super().__init__()
        self.dim = dim
        self.max_wavelength = max_wavelength
        # Learnable frequency multipliers for wall-clock time
        inv_freq = 1.0 / (max_wavelength ** (torch.arange(0, dim, 2).float() / dim))
        self.register_buffer("inv_freq", inv_freq)

    def forward(self, timestamps):
        # timestamps: [B, L] absolute Unix timestamps or cumulative deltas
        # Output: sin and cos components of shape [B, L, dim]
        device = timestamps.device
        # Outer product: [B, L, 1] * [dim/2] -> [B, L, dim/2]
        sinusoid_inp = torch.einsum("bl,d->bld", timestamps, self.inv_freq)
        emb = torch.cat((sinusoid_inp.sin(), sinusoid_inp.cos()), dim=-1)
        return emb # Transferred to match query/key dimensions in your temporal transformer

class AdvancedTransactionFoundationModel(nn.Module):
    def __init__(self, d_model, num_categories, num_numerical, total_attributes):
        super().__init__()
        self.d_model = d_model
        
        # 1. Feature Projections
        self.cat_embeddings = nn.Embedding(num_categories, d_model)
        self.num_projection = nn.Linear(num_numerical, d_model)
        
        # 2. Intra-Transaction Learned Attribute Identifiers
        self.attribute_slots = nn.Embedding(total_attributes, d_model)
        
        # 3. Dynamic Attention Stacking Aggregator
        self.intra_tx_attn = nn.MultiheadAttention(embed_dim=d_model, num_heads=8, batch_first=True)
        self.tx_norm = nn.LayerNorm(d_model)
        
        # 4. Temporal Sequence Backbone
        encoder_layer = nn.TransformerEncoderLayer(d_model=d_model, nhead=8, dim_feedforward=d_model*4, batch_first=True)
        self.temporal_backbone = nn.TransformerEncoder(encoder_layer, num_layers=6)

    def forward(self, cat_feats, num_feats, attribute_mask, cumulative_times):
        # cat_feats: [B, L, Num_Cats]
        # num_feats: [B, L, Num_Nums]
        # attribute_mask: [B, L, Total_Attributes] (0 for missing/null attributes, 1 for present)
        # cumulative_times: [B, L] (Timestamps used for time-aware relative positioning)
        
        B, L, _ = cat_feats.shape
        
        # Step 1: Align dimensions
        c_emb = self.cat_embeddings(cat_feats) # [B, L, Num_Cats, d_model]
        n_emb = self.num_projection(num_feats).unsqueeze(2) # [B, L, 1, d_model]
        
        # Combine into an isolated attribute block per transaction
        raw_stack = torch.cat([c_emb, n_emb], dim=2) # [B, L, Total_Attributes, d_model]
        
        # Step 2: Inject Intra-Tx Structural Positions (Which field is which)
        attr_positions = self.attribute_slots(torch.arange(raw_stack.size(2), device=raw_stack.device))
        stacked_inputs = raw_stack + attr_positions.unsqueeze(0).unsqueeze(0) # Broadcasting over B and L
        
        # Reshape to treat every transaction globally independent during token generation
        flat_stacked = stacked_inputs.view(B * L, -1, self.d_model) # [B*L, Total_Attributes, d_model]
        flat_mask = attribute_mask.view(B * L, -1) # [B*L, Total_Attributes]
        
        # Convert binary token mask to causal/additive attention mask (-inf for invalid fields)
        # MultiheadAttention uses key_padding_mask where True means 'ignore'
        attn_padding_mask = (flat_mask == 0) 
        
        # Step 3: Run Dynamic Attention Pooling over populated slots
        attn_out, _ = self.intra_tx_attn(
            flat_stacked, flat_stacked, flat_stacked, 
            key_padding_mask=attn_padding_mask
        )
        
        # Average pool over only the unmasked attributes
        # Prevent division by zero for completely unpopulated entries using clamp
        div_mask = flat_mask.sum(dim=-1, keepdim=True).clamp(min=1)
        pooled_tx_flat = (attn_out * flat_mask.unsqueeze(-1)).sum(dim=1) / div_mask
        
        # Reshape back to chronological sequence batches
        tx_tokens = pooled_tx_flat.view(B, L, self.d_model)
        tx_tokens = self.tx_norm(tx_tokens)
        
        # Step 4: Downstream Temporal Modeling 
        # Apply standard causal sequence padding mask for history length variations
        output_history = self.temporal_backbone(tx_tokens)
        
        return output_history
